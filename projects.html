Hole 19 Local Golf Shop Website

HTML · CSS · JavaScript · Netlify

Hole 19 is a professional mockup for a local golf shop in Springfield, Illinois, specializing in equipment sales and custom club fitting services. The site targets amateur to professional golfers and communicates a helpful, knowledgeable brand voice across five fully designed pages. A modern green aesthetic and clean layout make navigation intuitive for both casual browsers and serious buyers.

Key Features

  • Dark mode toggle
  • Five-page site structure
  • Club fitting scheduling
  • Business history stats
  • Gallery: studio, workshop, showroom

Retrospective

Looking back, I'd replace the static scheduling forms with a real backend integration — something like Netlify Forms or a lightweight API. I'd also build mobile responsiveness in from day one rather than retrofitting it, and add micro-interactions like transitions and hover states to better match the polished brand identity the project aimed for.

Checkout Form Website

HTML · CSS · JavaScript · Netlify

A front-end checkout form application built to practice structured form design, input validation, and responsive layout principles. The project simulates a realistic e-commerce checkout flow with a clean, user-friendly interface. It served as a focused exercise in building forms that are both functional and accessible.

Key Features

  • Client-side form validation
  • Billing, shipping, and payment sections
  • Responsive layout
  • Minimal, friction-free UI

Retrospective

If I were building this today, I'd integrate a payment library like Stripe Elements to make it production-ready and use React Hook Form for cleaner state management. I'd also go deeper on accessibility — ensuring all fields have proper labels, ARIA attributes, and full keyboard navigation support.

F1 Race Tracker

React · HTML/CSS · JavaScript · Firebase · MongoDB · Claude AI · Netlify

A React application built for Formula 1 fans to track, log, and organize their race viewing experiences in one place. Users can save favorite teams and drivers, log race results with personal ratings and notes, and connect with other fans through a built-in community board. The project evolved from a localStorage prototype into a full-stack application backed by a real database.

Key Features

  • Save favorite teams and drivers
  • Log races with ratings and notes
  • Filter by season
  • Fan community board
  • Profile and account management

Retrospective

The biggest thing I'd change is committing to a real database from day one instead of starting with localStorage. The migration mid-project cost significant time and introduced the bugs I'm still resolving — like race history appearing when logged out and duplicate race entries. I'd also set up global state management early on, either Zustand or Redux, to avoid the inconsistencies that came from passing state around manually.

PGA Golf Live Leaderboard

React · RapidAPI · OpenWeather API · HTML/CSS · JavaScript · Netlify

A React dashboard that displays live PGA Tour standings alongside real-time weather conditions for the tournament location, all in a single interactive view. The project demonstrates practical external API integration — combining two live data sources into a cohesive UI for golf fans. It's a hands-on example of working with asynchronous data and managing multiple API responses in React.

Key Features

  • Live PGA Tour standings
  • Real-time tournament weather
  • Dual-API dashboard
  • Async fetch with loading states

Retrospective

I'd add more robust error handling for when either API is rate-limited or unavailable — right now a failed fetch isn't handled gracefully. I'd also implement a lightweight caching layer to reduce redundant calls on re-renders, and prioritize a mobile-first layout so fans can check scores from their phones courtside.

Local File Reader — MCP Server

MCP (Model Context Protocol) · Node.js · Claude · GitHub

A custom Model Context Protocol (MCP) server that gives Claude the ability to read text files directly from a local directory — no manual copy-pasting required. Built as my first MCP tool, it was a hands-on exploration of how to extend Claude's capabilities beyond the chat interface and into a developer's actual file system. It's especially useful for tasks like summarizing notes, analyzing local datasets, or reviewing logs in context.

Key Features

  • Reads local files via MCP
  • Native Claude integration
  • No copy-paste required
  • Simple GitHub setup

Retrospective

I'd expand support beyond plain text to include JSON parsing, CSV row access, and Markdown rendering for richer data extraction. I'd also add directory listing so Claude can browse available files before selecting one, improve error messages for misconfigured paths, and package it as a clean npm install experience to lower the barrier for other developers.

Tip Calculator

React 18 · Vite · Vanilla CSS · GitHub Pages

A small, focused React app that calculates tip amounts and splits the bill for a group. I built this as a deliberate portfolio exercise — something constrained enough to finish in an afternoon but complex enough to require real decisions about state architecture, controlled input patterns, and user-friendly validation. The goal wasn't to build something novel; it was to build something common and do it carefully.

Key Features

  • Preset tip buttons (15%, 18%, 20%, 25%)
  • Custom tip input with preset sync
  • Bill splitting with per-person totals
  • Blur-triggered field validation
  • Empty state before input
  • One-way data flow, stateless children

Retrospective

The calculator itself is simple — what I was actually practicing was making decisions I could defend. I chose blur validation because surfacing errors mid-type is hostile UX. I centralized validation as pure functions so I could reason about them in isolation. I extracted ErrorMessage as its own component not because one was hard to write, but because three components sharing identical markup is exactly where extraction starts paying off. If I were scaling this: TypeScript, a form library, a custom hook to separate logic from layout. At this size, those would have been overhead without payoff — knowing where that line is matters as much as knowing the tools.
